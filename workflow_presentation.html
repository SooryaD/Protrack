Protrack Architecture
Overview

System Overview & Security Architecture

Protrack is a comprehensive, asynchronous state-machine designed to handle the intricate workflow of University-level project evaluations. It abandons paper trails in favor of a strictly structured role-based access pipeline.

Technical Stack

  • Frontend: React.js, Vite, Context API for persistent session management.
  • Backend: Node.js, Express.js REST API layer.
  • Authentication: JSON Web Tokens (JWT) & bcrypt securely hashing passwords.
  • State Transitions: Enforced backend middleware verifies roles before allowing a project to jump to the next grading stage.

Guiding Principles

  • Zero Data Loss: Missing fields automatically fallback or prevent DB commits.
  • Capacity Constraints: The system actively prevents guides from accepting more students than their allowed quota (default 2/guide).
  • Strict Sequence: A student cannot skip to Milestone 2 without passing Milestone 1. Admins cannot review before a Guide approves.

Actors

The 3 Core Actors (Role-Based Access)

1. The Student

The instigator of the pipeline. Their UI is strictly limited to their own project ticket.

  • Self-registers and verifies pre-authorized Roll Numbers.
  • Initiates Guide Requests.
  • Submits Project Abstracts, domains, and technologies.
  • Uploads Final Completed Reports.

2. The Staff Guide

The academic mentor. They are governed by an active "Student Capacity Limit".

  • Approves/Rejects incoming mentorship requests dynamically.
  • Filters and approves the initial project title and abstract.
  • Acts as the primary grader for the Midterm (R1) and Final (R2) milestone reviews.
  • Authorizes the final project completion for Viva-Voce.

3. System Admin & CSC

The overarching university authority and Central Steering Committee.

  • Maintains massive overhead visibility of all project statuses.
  • Manages the raw Capacity quotas for individual staff guides.
  • Performs the crucial CSC Audit: Determining if the project scope is adequate for a master's degree before development begins.

Phase 1 & 2: Initiation

Phase 1 & 2: Guide Matchmaking & The Proposal

Phase 1: Guide Matchmaking

Before any project work begins, the student must secure a guide. This prevents "orphan" projects from entering the database.

  • The student views a real-time list of all Guides and their current free slots (e.g., Capacity 1/2).
  • The student submits a REQUEST PENDING ticket.
  • The Guide reads the request. If they decline, the student is notified to try another Guide. If they accept, the system instantly sets the student's assignedGuideId, binding them permanently.

Phase 2: Project Proposal formulation

The student drafts their application. The system generates a unique tracking ID (e.g., EDU-PRJ-2026-004) ensuring tracking continuity.

  • Title & Abstract: The core of the idea.
  • Tech Stack & Domain: E.g., 'React, Node, Python' in 'Cloud Computing'.
  • Status becomes TITLE_PENDING. Only the assigned Guide can view and approve this. Once Guide approves, it transitions to TITLE_APPROVED.

Phase 3: Administrative Audit

Phase 3: The Central Steering Committee (CSC) Audit

Once a Guide approves the initial proposal, development cannot legally start. It enters a holding pattern awaiting University-level clearance by the Admin (CSC pane).

CSC Decision Matrix

✅ Approved

The scope is deemed sufficient for a university degree. The project status shifts to CSC_APPROVED.

Action: The student is permitted to begin development and system design.

↩ Scope Modification

The idea is good, but too simple or overly complex. Admin must provide a strict written reason.

Action: Ticket bounces back to Student. They must edit the abstract and resubmit.

❌ Not Approved

The submitted project has failed the audit entirely (e.g., plagiarism, inappropriate domain).

Action: The concept is killed. The student must start over with a fresh proposal.

Phase 4 & 5: Grading & Completion

Phase 4 & 5: Milestone Reviews and Finalization

Once CSC Approved, the student builds the software. The Staff Guide conducts two highly structured, math-based grading reviews.

Review 1 (First Review) R1

Evaluates initial problem-solving.

  • Problem Definition & Scope (20)
  • Literature Review / Feasibility (20)
  • Novelty / Creativity (10)
  • Detailed Design / Constraints (20)
  • Methodology (30)

System mathematically sums out of 100, then normalizes to a 20-point scale for the university ledger.

Review 2 (Second Review) R2

Evaluates execution and outcomes.

  • System Design / Work Flow (20)
  • Modules Implementation (50)
  • Data Set / Outputs (20)
  • Contribution (10)

System mathematically normalizes to a 20-point scale. Status becomes SECOND_REVIEW_DONE.

Phase 5: The Final Project State

The student clicks "Submit Final Report". The guide reviews the physical/digital documents, checks the code for plagiarism, and triggers the final system state: PROJECT_COMPLETED. The student is now cleared to graduate and attend Viva-Voce.
